인프라 전략: "레고 블록처럼" 코드로 조립하고 자동화하기 🧱

저희는 서버, 데이터베이스 같은 인프라를 더 이상 손으로 하나씩 만들지 않습니다.
마치 레고 설명서를 보듯, '코드'를 통해 전체 시스템을 조립하고 완전 자동화합니다.

저희 인프라 파트의 핵심 철학은 바로 '레고 블록처럼 코드로 조립하고 자동화하기'입니다.

서버 한 대, 데이터베이스 하나를 마우스로 클릭해서 만드는 시대는 지났습니다. 저희는 모든 인프라 구성을 '코드'로 정의하고, 이 코드들을 레고 블록처럼 조립하여 전체 시스템을 완성시킵니다.

지금부터 왜 이런 방식을 선택했는지, 그 재료가 된 AWS와 조립 도구인 Terraform 이야기를 들려드리겠습니다.

Why? ① 재료: 왜 AWS를 선택했는가?

압도적인 안정성과 성숙도 때문입니다. 저희는 '직접 만드는 것'보다 '잘 만들어진 것을 쓰는 것'에 집중했습니다.

  • 관리형 서비스 (RDS, S3): 복잡한 DB 설치, 백업, 보안 패치나 파일 서버 운영 부담에서 벗어나 오직 개발에만 집중할 수 있었습니다.
  • 유연한 확장성 (Auto Scaling): 지금은 작은 규모로 시작하지만, 사용자가 늘어났을 때 버튼 몇 번으로 서버를 증설할 수 있는 확장 가능성이 매력적이었습니다.
  • 풍부한 생태계와 문서: 문제가 발생했을 때 해결책을 찾기 쉽고, 전 세계적으로 가장 많은 사용 사례를 보유하고 있어 신뢰할 수 있었습니다.
첫 번째, 왜 재료로 AWS를 선택했을까요? 바로 압도적인 안정성과 성숙도 때문입니다.

저희는 직접 데이터베이스 서버를 설치하고 관리하는 대신, AWS가 제공하는 관리형 서비스인 RDS를 사용했습니다. 덕분에 복잡한 DB 설치, 백업, 보안 패치 같은 운영 부담에서 벗어나 오직 개발에만 집중할 수 있었습니다. 파일 저장 역시 S3라는 강력한 서비스를 활용했습니다.

또한, 지금은 작은 규모로 시작하지만, 사용자가 늘어났을 때 Auto Scaling Group 같은 기능을 통해 손쉽게 서버를 증설할 수 있는 유연한 확장 가능성이 매력적이었습니다.

Why? ② 조립 도구: 왜 Terraform을 선택했는가?

AWS라는 좋은 재료를 매번 손으로 조립하는 것은 비효율적입니다. 저희는 코드형 인프라(IaC) 방식, 그중에서도 Terraform을 채택했습니다.

  • 재사용성과 일관성: VPC, EC2, RDS 등 각 기능을 독립적인 모듈(레고 블록)로 만들어, 언제 어디서든 오차 없이 동일한 구성의 인프라를 신속하게 구축할 수 있습니다.
  • 형상 관리 및 협업: 모든 인프라 구성이 Git에 코드로 남기 때문에, 누가, 언제, 무엇을 변경했는지 추적하기 쉽고 팀원 간의 협업이 원활해집니다.
  • 완전 자동화의 기반: 코드로 관리된다는 것은, GitHub Actions 같은 CI/CD 도구와 연동하여 사람의 실수 없이 배포 전 과정을 자동화할 수 있다는 의미입니다.
두 번째, 그렇다면 이 AWS라는 재료들을 어떻게 관리했을까요? 저희는 Terraform을 사용한 코드형 인프라(IaC) 방식을 채택했습니다.

가장 큰 장점은 '재사용성'과 '일관성'입니다. 저희는 VPC, EC2, RDS 등 각 기능을 독립적인 모듈, 즉 레고 블록으로 만들었습니다. 이렇게 하면 개발 환경이든 운영 환경이든 항상 동일한 구성의 인프라를 오차 없이, 그리고 매우 신속하게 구축할 수 있습니다.

또한 모든 변경사항이 코드로 남기 때문에 버전 관리가 가능하고, 궁극적으로는 GitHub Actions와 연동하여 인프라 변경까지 완전히 자동화하는 기반이 됩니다.

잠깐, CI/CD가 뭔가요? 🤔

CI/CD는 "코드를 제품으로 만드는 자동화 공장"에 비유할 수 있습니다.
개발자가 '코드'라는 원재료를 넣으면, 테스트, 빌드, 배포 과정이 자동으로 처리되어 사용자에게 '제품'이 전달됩니다.

CI (Continuous Integration)

지속적인 통합: "부품 조립 및 품질 검사"

여러 개발자의 코드를 합칠 때마다 자동으로 빌드(조립)하고 테스트(검사)하여 코드의 품질과 안정성을 보장하는 단계입니다.

CD (Continuous Deployment)

지속적인 배포: "완제품 포장 및 출시"

CI 단계를 통과한 결과물을 사용자가 실제로 사용하는 서버(운영 환경)에 자동으로 배포(출시)하는 단계입니다.

방금 '자동화'라는 말을 썼는데, 본격적인 설명에 앞서 CI/CD라는 개념을 잠깐 짚고 넘어가겠습니다. 아마 많이 들어보셨을 텐데요, 어렵게 생각할 것 없이 '코드를 제품으로 만드는 자동화 공장'이라고 생각하시면 쉽습니다.

먼저 CI, 지속적인 통합입니다.
이건 공장의 '부품 조립 및 품질 검사' 라인입니다. 여러 개발자가 만든 코드 조각(부품)들을 하나로 합칠 때마다, 이 코드가 잘 조립(빌드)되는지, 그리고 문제는 없는지(테스트) 자동으로 검사하는 과정입니다. 이 덕분에 코드의 품질을 항상 높게 유지할 수 있습니다.

다음은 CD, 지속적인 배포입니다.
이건 '완제품 포장 및 출시' 라인입니다. 품질 검사를 통과한 완제품(소프트웨어)을 사람이 직접 옮기는 게 아니라, 컨베이어 벨트가 자동으로 사용자에게 전달(배포)하는 과정입니다.

저희 프로젝트에서는 GitHub Actions가 바로 이 CI/CD 공장 역할을 합니다. 이제 이 공장이 저희 인프라를 어떻게 자동으로 만들어내는지 보여드리겠습니다.
